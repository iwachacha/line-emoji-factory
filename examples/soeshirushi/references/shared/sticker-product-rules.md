# スタンプ商品ルール

このファイルは、ブランドを実際のLINEスタンプ商品へ落とす際の判断基準を扱う。
商品品質に関する項目は、絵文字固有の判断とは分けてこのファイルで扱う。

## 公式仕様との接続

- 技術仕様と販売運用事実は `rules/line-platform-baseline.md` を前提にする。
- このファイルでは、公式制作ポイントをスタンプの商品判断へ翻訳して扱う。
- 通常スタンプを対象にする。アニメーション、カスタム、メッセージ、BIG、ポップアップ、エフェクトの各スタンプは、別途仕様確認と packaging 契約ができるまで対象外にする。

## 商品層で必ず見る項目

- `視認性`
- `即読性`
- `会話投入性`
- `反応完結性`
- `再使用性`
- `差分明確性`
- `セット整合性`
- `感情レンジ`
- `量産性`
- `商品転換力`

## 商品設計の原則

- 1スタンプ単体で、送信意図と反応温度が読める。
- 日常会話、挨拶、確認、謝意、軽い感情表現など、繰り返し使う場面がある。
- スタンプは文中補助ではなく、会話に1つの発話ブロックとして挿入される前提で設計する。
- 顔、ポーズ、文字、効果線のどれが主情報かを固定する。
- 横長すぎる構図、全身が細く小さくなる構図、淡色だけの構図で視認性を落とさない。
- セット内に実質重複を作らない。
- 色違いだけ、微差分だけで差分を水増ししない。
- ブランドの核が、表情、ポーズ、短文、モチーフのいずれかへ自然変換されている。
- 初期セット個数は `公式最小数だから` ではなく、商品コンセプト、差分の軸、再使用場面の幅に見合う根拠で決める。
- release / set ごとに `この商品を買う理由` を一言で言える状態にする。これが弱いと `セット整合性` と `商品転換力` が落ちる。

## 公式制作ポイントの運用化

- 余白は10px前後を目安にし、上下左右のバランスを崩さない。
- 文字入りスタンプは、小表示のプレビューでも読める文字数と線幅に抑える。
- 文字だけに寄せすぎず、表情、ポーズ、モチーフでブランド固有性を残す。
- 表情や反応差分は、サムネイルでも分かるほど大きくする。
- 装飾は主情報を助ける範囲にとどめ、情報量で読ませない。
- セット上部には高頻度用途を置く。
- 構図やポーズに変化を入れ、パッケージ全体を単調にしない。

## 構造分類ごとの最低合格条件

### 単体反応型

- 各スタンプが単体で発話として成立する。
- 高頻度反応語彙に、ブランド固有の温度差がある。

### キャラクター発話型

- キャラクターの人格と口調がぶれない。
- 顔だけ、全身だけ、文字だけに偏らず、表情と発話の対応が読める。

### モチーフ発話型

- モチーフが小表示でも識別できる。
- モチーフの説明より、会話での使い道が先に読める。

### テキスト主導型

- 文字の可読性を最優先する。
- 文字が多すぎてスタンプ画像ではなく告知画像に見えない。
- 文字だけで既存定型文と差が出ない場合は、造形か口調の差分を足す。

### 混合型

- 主構造が明示されている。
- 主構造側の最低合格条件を満たしている。

## ブランド主型ごとの最低合格条件

### キャラクター

- 同一ブランド内で人格と反応温度がぶれない。
- 表情差分、ポーズ差分、短文差分が互いに補強している。

### モチーフ

- モチーフが会話反応へ変換されている。
- 物体や景色だけで終わり、日常会話で使いにくい状態を避ける。

### 記号

- 記号が感情や会話機能へ変換されている。
- 企業ロゴや広告記号に寄りすぎない。

### 図形・抽象

- 造形文法が揃っている。
- 抽象性が高くても、反応や用途が無に落ちない。

### デコ文字

- 文字が主役であることを崩さない。
- 可読性とスタンプらしい発話完結性を両立する。

## 補助タグ付きの最低合格条件

### 実用品・機能

- 高頻度シーンが明示できる。
- 既存の汎用表現と比べた実用優位点がある。

### 季節・文化

- 使用期間または使用文脈が定義されている。
- シーズン外に使えない場合は、通年派生方針がある。

## ブランド固有基盤への接続

- `Design Ready` の後、制作実装へ進める場合は `workflows/production-pipeline-workflow.md` で工程責務を固定する。
- `Design Ready` の後、制作実装へ進める場合は `templates/brand/brand-production-brief-template.md` を埋める。
- 初期 release / set 設計は `templates/release/release-spec-template.md` を埋める。
- release / set 単位の引き継ぎは `templates/release/production-handoff-template.md` を埋める。
- 提出前と継続改善は `workflows/quality-control-workflow.md` と `templates/qa/release-checklist-template.md`, `templates/qa/quality-ledger-template.md` を使う。
- `brand-production-brief` では、推奨パッケージタイプ、初期個数、差分配列、余白方針、タグ方針、審査注意、工程責務を固定する。

## 判定の目安

- `Revise`
  - 用途はあるが、視認性・差分明確性・セット整合性のいずれかが弱い。
  - 初期セット個数の根拠や release 固有コンセプトが弱く、商品として買う理由が薄い。
- `Design Ready`
  - セット構成、差分設計、量産方針、rough stage と仕上げ stage の接続方針、初期 release 設計の筋まで明文化できる。
