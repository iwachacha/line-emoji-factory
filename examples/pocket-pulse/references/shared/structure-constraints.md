# 構造成立性ルール

このファイルは、LINE絵文字 / スタンプブランド判断の入口条件を定義する。
ブランドや商品を論じる前に、まずここを通す。

## 公式事実の前提
- 技術仕様、パッケージ数、画像形式、アニメーション制約は `rules/line-platform-baseline.md` を正本にする。
- 構造判断では、先に対象 item type を仮置きする。未確定なら `static-emoji` と `static-sticker` の両方で成立可能性を見る。
- 絵文字構造では、`単体送信時`、`インライン表示時`、`複数絵文字連続時` の3状態を意識する。
- スタンプ構造では、`単体送信時`、`トークの発話ブロックとしての表示時`、`セット内で連続使用された時` の3状態を意識する。

## 基本前提
- LINE絵文字は、`1画像単位` で意味が成立する前提で考える。
- LINEスタンプは、`1画像単位` で発話または反応が完結する前提で考える。
- テキストと一緒に使うときは、小さくインライン表示されても意味が落ちないことを前提にする。
- 単体送信でも破綻しすぎないことを前提にする。
- 複数絵文字で見た目を作る場合も、各パーツは独立した絵文字として成立している必要がある。
- デコ文字は、自由装飾ではなく `固定対応文字セット` として扱う。
- テキスト絵文字は、他絵文字とつながりすぎない余白運用を前提にする。
- テキスト入りスタンプは、文中補助ではなく発話として読める量と可読性に抑える。

## そのまま通してはいけない発想
- 任意文字列全体を、1つの絵文字で囲む。
- 任意文字列に、下線・背景帯・囲み枠を後付けする。
- 絵文字の中に別の文字が自動で入る前提で考える。
- 文の長さや改行位置によって意味成立が変わる設計に頼る。
- 特殊なレイアウトが成立しないと意味が通らない。
- 単体では意味がなく、配置依存だけで成立する。
- スタンプを文中に差し込む前提で、単体発話としての弱さを隠す。

## 構造分類
### 単体反応型
- 1画像ごとに反応・意味・用途が独立して成立する型。

### 組み合わせ型
- パーツを並べると狙った演出が強まるが、各パーツ単体でも最低限の意味が残る型。
- `絵文字間に余白が入らない` というLINEの表示事実を利用してよいが、単体使用不能を正当化してはいけない。

### スタンプ発話型
- 1画像ごとに発話・反応・用途が独立して成立する型。
- 表情、ポーズ、短文のどれが主情報かを明記する。
- 日常会話で使用しにくい物体・景色だけの案は、先に発話化または反応化する。

### デコ文字型
- 固定対応文字セットの可読性が主役になる型。
- ブランド側でも扱う場合は、`構造分類がデコ文字型であること` を前提にする。
- 文字として読めることが、装飾より優先される。

### 混合型
- 複数構造が混在する型。
- `主構造` と `混合理由` を必ず明記する。
- 逃げ道として使わない。

## 構造チェック
1. 1画像単位で意味があるか。
2. 単体送信でも破綻しすぎないか。
3. 小表示またはスタンププレビューで主要要素が読めるか。
4. 任意文字列への後付け装飾前提になっていないか。
5. 組み合わせ型なら、各パーツ単体の弱さを放置していないか。
6. デコ文字型なら、固定対応文字セットとして可読性が維持できるか。
7. 混合型なら、主構造を言語化できるか。
8. アニメーションを使う場合でも、1フレーム目だけで意味が読めるか。
9. スタンプなら、発話ブロックとして送る理由があるか。

## Fail時の扱い
- 構造でFailした案は、そのままブランド評価や商品化議論へ進めない。
- `workflows/transformation-workflow.md` に沿って、成立形への変換案を先に出す。
- 変換後もFailする場合だけ、切る判断を行う。

## 他ルールとの関係
- 評価語彙と判定状態は `rules/evaluation-model.md` を使う。
- ブランド主型や補助タグは `rules/brand-taxonomy.md` を使う。
