# 継続改善ルール

このファイルは、監査結果、市場観測、制作事故、審査差し戻し、skill 運用上の詰まりを、
単発修正で終わらせず再利用可能な学習へ変える判断基準の正本である。

## 境界
- 判断順序は常に `構造 → ブランド → 商品`。
- 単発の感想だけで rule を増やさない。
- brand 固有問題を、すぐ factory 共通問題へ昇格させない。
- 逆に再発問題を brand 側だけで抱え込まない。
- work 完了は、owner file 更新、記録更新、remote push 完了まで含む。

## 改善単位
- `Observation`
  - 何が起きたかを 1 文で書く。
- `Diagnosis`
  - どのレイヤーで壊れたか、根本原因は何かを切り分ける。
- `Intervention`
  - どの owner file を直すか、または何を試すかを決める。
- `Verification`
  - 次にどこで効き目を見るかを決める。
- `Promotion`
  - brand 固有で止めるか、factory へ上げるか、skill を育てるかを決める。

## 学習源
- 創出ブランドの監査
- 生成された絵文字、rough board、handoff、final asset の監査
- 審査差し戻し、公開後の drift、metadata 事故
- 市場調査つきのアイデア探索
- prompt、workflow、template、skill の運用詰まり
- 同じ依頼が繰り返し発生すること自体

## スコープ分類
- `brand local`
  - 単一ブランド、単一 release、単一 set、単一 handoff で閉じる問題。
- `factory common`
  - 2 release 以上、2 ブランド以上、または scaffold 不備で再発しうる問題。
- `skill maintenance`
  - 入口の分岐、読む順序、出力枠、owner file 参照に継続的な詰まりがある問題。
- `monitor only`
  - まだ規則化や改修より観測が先の問題。

## factory へ昇格させる条件
次のどれかに当たるときは、brand 側だけで閉じず factory owner file を更新候補にする。
- 同種の問題が 2 release 以上で再発した。
- 同種の問題が 2 ブランド以上で再発した。
- script や template の不足が原因だった。
- 現行 workflow では再発抑止が弱い。
- 市場調査や監査で同じ blind spot が繰り返し見えた。

## skill を育てる条件
次のどれかに当たるときは、既存 skill の改修または新設を検討する。
- 同じ読む順序や判断順が毎回口頭補足で必要になる。
- 同じ出力骨格や監査観点が 3 回以上繰り返し必要になる。
- 既存 2 skill 以上にまたがる運用が常態化し、入口が曖昧になっている。
- workflow や template だけでは、実務入口の抜けが埋まらない。

## 軽量運用原則
- 学習台帳には、再利用可能な論点だけを残す。
- 1 回で閉じた軽微なノイズは、長文で残さない。
- 閉じた論点が増えすぎて active な論点を埋めるなら、要約して圧縮する。
- 同じことが複数ファイルに書かれているなら、owner file に寄せて他方は削る。
- skill が重くなったら、足す前に `削る / まとめる / 役割を戻す` を先に検討する。
- 改善の仕組み自体が重いと観測されたら、その仕組みを改善対象に含める。
- 具体会話検証は、決定を動かした 6〜12 ケースだけ残し、枝葉の全ログは残さない。
- retrospective は milestone ごとに 1 本を原則にし、旧版は要約へ畳む。

## 共通基盤へ上げる前の最小証拠
- `factory common` へ昇格させる改善は、少なくとも 2 つの具体証拠を持つ。
- 証拠は、2 brand / 2 release の再発、または 1 release 内でも 2 つ以上の明確な使用シーン差分のどちらかでよい。
- 具体会話検証で `句読点 / 既存記号 / 既存絵文字` と大差が出ない場合は、商品改善より先に `買う理由` を再点検する。

## 最新情報の扱い
- 現行市場の傾向、人気、飽和、空白を語るときは都度観測し、日付付きで扱う。
- 単発の市場観測だけで factory の恒久 rule にしない。
- 強い観測でも、まずは `monitor only` または限定改善から入る。
