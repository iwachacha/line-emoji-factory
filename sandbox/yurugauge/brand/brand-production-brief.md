# ゆるゲージ ブランド固有制作基盤

## 1. 基本情報
- ブランド名: `ゆるゲージ`
- 参照するブランド設定: `brand/brand-setting.md`
- 判定状態: `Design Ready`
- 商品化の目的: 状態共有と短い返答を、低圧で繰り返し使えるLINE絵文字にする。
- 商品コンセプト: 進捗と返事を1文字ゲージで軽く返す。
- ブランド内での役割: 初回コア16で視覚文法と高頻度語彙を固定する。

## 2. 公式仕様の確定
- 商品種別: `静止絵文字`
- パッケージタイプ: `絵文字`
- 初期セット個数: `16`
- 初期セット個数の根拠: `8` では状態共有の主要語彙が不足し、`40` では敬語テキストの混雑へ寄りすぎる。`16` なら高頻度返答、進行、保留、締めを1商品で示せる。
- 必須画像: トークルームタブ画像 `96 x 74 px` を1個、コンテンツ画像 `180 x 180 px` を16個。
- 画像形式: `PNG`
- 画像サイズ: コンテンツ `180 x 180 px`、タブ `96 x 74 px`
- 容量制約: 画像1個 `1MB以下`、ZIP `20MB以下`
- 背景透過: 必須
- メタデータ制約: クリエイター名50文字以内、タイトル40文字以内、説明文160文字以内、コピーライト50文字以内かつ英数字のみ。metadata に絵文字、URL、告知文言を入れない。

## 3. 固定制作パイプライン
- rough stage の目的: 16個のゲージ量、ラベル、補助サインが重複せず読めるか確認する。
- rough stage で先に固定するもの: ゲージ外形、下部バー、中央ラベル、アクセント色の幅、状態サインの種類。
- rough board / 全体図の最小構成: 4 x 4 のセット一覧、タブ画像案、混同しやすい3組の比較。
- rough stage の成功条件: 各絵文字の単体意味、文中役割、セット内差分が説明できる。
- ClaudeDesign へ渡す単位: 1絵文字ごとの `用途 / ラベル / バー量 / 色 / 補助サイン / 注意点`。
- ClaudeDesign で仕上げる範囲: 小表示可読性、輪郭の太さ、文字の中心配置、バーの視認性、透明余白。
- 仕上げで変えてよいもの: 色の微調整、バー幅、補助サインの位置、文字の太さ。
- 仕上げで変えてはいけないもの: ラベル意味、ゲージ外形、状態差分、低圧トーン。
- 差し戻しルール: 構造後退は構造再評価、ラベル混同は release spec、視認性だけなら final asset 修正。

## 4. 表示前提
- 単体送信時の役割: 状態返答として単独で成立する。
- 文中使用時の役割: 短文の前後や末尾に置き、状態や進行度を補足する。
- 複数絵文字連続時の役割: `中 → 済` のような状態推移を2個まで示す。
- 余白方針: 公式推奨に合わせて外形は大きく使う。透明余白は上下左右14px前後に抑える。
- アウトライン方針: 10px相当の濃色アウトライン。小表示でも輪郭が残る太さ。
- 装飾方針: 補助サインは1個のみ。影は薄く、輪郭とバーを優先。

## 5. セット構成
- 最上段に置く高頻度絵文字: `001 見`, `002 了`, `003 中`, `004 済`
- 基本反応語彙: 見た、了解、対応中、完了、確認、要返信、共有、ありがとう、保留、あとで、急ぎ、締切、調整、未定、決定、休憩。
- 差分の軸: 返答 / 進行 / 保留 / 締切 / 調整 / 感謝。
- 主力枠と補助枠の配分: 主力12、補助4。補助は `礼`, `休`, `未`, `締` のように用途が明確なものだけ。
- バリエーション方針: ラベル + バー量 + 状態サインで差を立てる。色だけでは増やさない。
- 並び順方針: 上段は最頻出返答、2段目は確認と共有、3段目は保留と期限、4段目は調整と締め。
- 重複禁止ルール: `了 / 済 / 決`、`中 / 調`、`見 / 確 / 返` は用途差が説明できない場合に差し戻す。

## 6. stage 別制作ルール
- 初期 release spec の作成ルール: 16個すべてに単体意味、文中役割、連続使用メモを持たせる。
- rough stage で使う視覚 anchor: 太い角丸ゲージ、1文字ラベル、下部バー、状態サイン。
- rough stage で未確定にしてよい点: 色の微差、補助サインの細部、影の有無。
- ClaudeDesign に必ず渡す共通条件: 透明背景、180px、72dpi以上、RGB相当、1MB以下、太線、余白少なめ。
- 1絵文字ごとの完成条件: 64px相当まで縮小してラベルと状態が読める。
- ファイル命名 / バージョン方針: Upload files are `001.png` to `016.png` and `tab.png`. 作業版は `production/finals/release-001/` に置く。
- release handoff の作成ルール: individual specs を `production/handoffs/release-001-handoff.md` に固定する。

## 7. デコ文字・組み合わせ・アニメーション個別方針
- デコ文字可読性ルール: 該当なし。1文字ラベルは絵文字内記号として扱う。
- 組み合わせ使用ルール: 2個まで。状態推移として読む用途に限定する。
- 組み合わせ補助枠の扱い: 初期 release には入れない。
- 1フレーム目ルール: 静止絵文字のため該当なし。
- アニメーションの見せ方: 後続で検討する場合も、1フレーム目で意味が読めることを必須にする。
- フレーム / 再生時間 / ループ設計メモ: 該当なし。

## 8. 登録・販売運用
- タブ画像方針: ブランド核が分かる小型ゲージを1つ置き、中央に `進` を入れる。
- タイトル方針: `ゆるゲージ 進捗と返事16`
- 説明文方針: `確認中・対応中・完了など、仕事や日常の短文に進み具合と返事を軽く添える実用絵文字です。`
- コピーライト表記: `YURUGAUGE`
- サジェスト表示タグ候補: `了解`, `完了`, `確認`, `対応`, `返信`, `共有`, `締切`, `あとで`
- リリース前チェック: `qa/release-checklist.md` を通し、blocking が残る場合は提出しない。

## 9. 審査注意
- Hard NG 回避項目: 公式サイズ・透過・形式・個数を守る。他者IP、ロゴ、外部サービス名を使わない。
- Revise になりやすい項目: 画像内文字の誤字、metadata と絵文字内容のズレ、重複差分。
- Watch 項目: 1文字漢字が小表示で読めるか、仕事向けに寄りすぎないか、`了 / 済 / 決` の混同。
- 品質台帳へ残すべき論点: 小表示可読性、混同3組、metadata検索語の過剰化。
- 継続監査で確認する仮説: ゲージ量が買う理由として機能するか。

## 10. 量産・拡張
- 次に増やす差分: 予定調整、感情温度、家事タスク、チーム連絡。
- 季節派生の可否: 初回は不可。季節より通年状態共有を優先。
- 実用品派生の可否: 可。予定調整版と家庭タスク版が候補。
- プレミアム / 長期運用の見立て: 公開から180日以降、再使用率と検索流入が見えたら検討。
- 商品化・ライセンス拡張メモ: 汎用状態記号として小規模チームやコミュニティ向け展開余地あり。ただし企業ロゴ化は避ける。
- factory へ戻す改善候補: final asset manifest とPNG仕様検査の共通テンプレート化。
- 運用を重くしないためのルール: release ごとに混同3組だけを重点監査し、全項目を長文化しない。
